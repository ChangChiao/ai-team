## Product Design Position

This is not a lifestyle community app and not a generic SaaS dashboard.

The UI should feel like a serious collector marketplace:

- precise
- sortable
- trust-oriented
- photo-forward
- quiet
- low decoration
- built for repeated scanning

The product should not use a marketing-style hero as the first experience. The first screen should be the marketplace itself.

## Visual Direction

### Personality

Collector-grade, not playful toy-store.

Use the feeling of:

- a well-lit display case
- an auction catalog
- a parts inventory sheet
- a trusted trading desk

Avoid:

- purple SaaS gradients
- decorative blobs
- icon-in-circle feature grids
- oversized marketing cards
- generic "all-in-one marketplace" copy
- cute toy-like styling

### Typography

Use a real typeface, not a default system stack.

Recommended:

- Display / headings: `Sora` or `Space Grotesk`
- Body / UI: `IBM Plex Sans` or `Inter Tight`
- Numeric details: tabular numbers enabled where possible

Rules:

- Body text minimum 16px.
- Dense metadata may go to 13-14px only if contrast is high and it is not primary reading text.
- No negative letter spacing.
- Listing titles should wrap to two lines before truncating.

### Color

Use a restrained neutral base with one mechanical accent.

Suggested CSS tokens:

```css
:root {
  --color-bg: #f7f6f2;
  --color-surface: #ffffff;
  --color-surface-subtle: #efede7;
  --color-text: #171717;
  --color-muted: #68645c;
  --color-border: #d8d3c7;
  --color-accent: #0f766e;
  --color-accent-strong: #0b4f4a;
  --color-danger: #b42318;
  --color-warning: #a15c07;
  --color-success: #137333;
}
```

The palette should not become beige-only. Product photos provide most of the visual color; the UI should stay quiet.

### Shape And Texture

- Border radius: 6px for cards and inputs.
- Buttons: 6px radius.
- Photo frames: 4px radius.
- Borders over shadows.
- Use grid lines and separators sparingly, like a catalog.
- Avoid floating cards inside other cards.

## Screen Specs

### 1. Marketplace Home `/`

Purpose: show that this is an active model car marketplace, not a generic landing page.

First screen hierarchy:

1. Search for a model car.
2. Browse current listings.
3. See trust-oriented seller signals.

Layout:

```text
+------------------------------------------------------+
| Logo     Browse  Sell  Sellers              Account  |
+------------------------------------------------------+
| Find model cars from sellers with trade history       |
| [ Search brand, model, scale...             ] [Search]|
| Quick filters: Hot Wheels | Tomica | Mini GT | 1:64   |
+------------------------------------------------------+
| Latest listings                         View all      |
| [Listing] [Listing] [Listing] [Listing]               |
| [Listing] [Listing] [Listing] [Listing]               |
+------------------------------------------------------+
| Sellers with confirmed transactions                  |
| Seller row | confirmed count | active listings        |
+------------------------------------------------------+
```

No hero illustration. No feature grid. The product is the inventory.

Empty state:

- If no listings exist, show seller acquisition state:
  - Heading: `Be the first seller with a public model car inventory`
  - Body: `List 5 cars, share your seller page, and start building confirmed transaction history.`
  - Primary action: `Create seller profile`

### 2. Browse Listings `/listings`

Purpose: help buyers scan, filter, and compare fast.

Desktop layout:

```text
+----------------------+--------------------------------+
| Filters              | Results header                 |
| - Search             | 128 listings В· newest first     |
| - Brand              | [Sort] [View: grid/list]        |
| - Scale              |--------------------------------|
| - Mode               | Listing grid                    |
| - Status             | 3-4 columns                     |
+----------------------+--------------------------------+
```

Mobile layout:

```text
+--------------------------------+
| Search                         |
| [Filter] [Sort]                |
+--------------------------------+
| Listing card                   |
| Listing card                   |
| Listing card                   |
+--------------------------------+
```

Filter drawer on mobile. No permanent sidebar on small screens.

Listing card hierarchy:

1. Photo
2. Title
3. Price / trade mode
4. Brand, scale, condition
5. Seller confirmed transaction count
6. Status badge

Card spec:

```text
+------------------------------+
| photo, fixed 4:3             |
+------------------------------+
| Mini GT Nissan Skyline...    |
| NT$1,200 В· Sale or trade     |
| Mini GT В· 1:64 В· Box: Good   |
| Seller: Aki Models В· 18 tx   |
| [Available]                  |
+------------------------------+
```

Card rules:

- Fixed image aspect ratio to avoid layout shift.
- Status badge top-left on photo.
- Sold listings are visually muted and not mixed into default browse.
- Missing price should show `Trade only` or `Ask seller`, never blank.

Empty state:

- Heading: `No matching cars yet`
- Body: `Try fewer filters, or save this search once alerts exist.`
- Primary action: `Clear filters`
- Secondary action: `Browse latest`

### 3. Listing Detail `/listings/:id`

Purpose: let buyer decide whether the item and seller are trustworthy enough to contact.

Desktop layout:

```text
+------------------------------------------------------+
| Breadcrumb: Listings / Mini GT / Nissan              |
+-----------------------------+------------------------+
| Photo gallery               | Title                  |
| main image                  | Price                  |
| thumbnails                  | Status                 |
|                             | Contact seller         |
+-----------------------------+ Share link             |
| Model details               | Seller trust panel     |
| condition, box, defects     | confirmed tx count     |
| delivery, location          | joined date, profile   |
+-----------------------------+------------------------+
```

First visible items:

1. Main photo.
2. Title.
3. Price or trade mode.
4. Seller trust panel.
5. Contact action.

Trust panel:

```text
Seller
Aki Models
18 confirmed transactions
12 active listings
Joined Mar 2026
[View seller profile]
```

Use precise language:

- `18 confirmed transactions`
- not `trusted seller`
- not `verified seller` unless verification exists

Defects and condition must be visible without expanding a hidden section. Buyers should not hunt for risk information.

Error state:

- If listing is removed: `This listing is no longer available`
- Show seller profile link if still public.
- Suggest browsing similar brand/scale.

### 4. Seller Profile `/sellers/:slug`

Purpose: make seller trust inspectable.

Hierarchy:

1. Seller identity.
2. Confirmed transaction count.
3. Active listings.
4. Sold history.
5. External social proof if provided.

Layout:

```text
+------------------------------------------------------+
| Seller name                         Share profile    |
| 18 confirmed transactions В· 12 active listings        |
| Location В· Joined date В· optional Facebook link       |
+------------------------------------------------------+
| Active listings                                      |
| [Listing] [Listing] [Listing] [Listing]               |
+------------------------------------------------------+
| Confirmed transaction history                         |
| Mini GT Nissan Skyline В· confirmed May 2026           |
| Tomica Limited Vintage... В· confirmed Apr 2026        |
+------------------------------------------------------+
```

Important trust rule:

Confirmed transaction history should show enough to be useful, but not expose buyer private data. Show listing title, transaction month, and transaction type. Do not show buyer email.

Empty state:

- New seller with no confirmed transactions:
  - `No confirmed transactions yet`
  - `This seller can build history when buyers confirm completed transactions.`
  - If owner viewing own page: `Record your first completed transaction`

### 5. Seller Dashboard `/dashboard`

Purpose: operational workspace, not decorative dashboard.

Desktop layout:

```text
+------------------+-----------------------------------+
| Dashboard nav    | Today                             |
| - Listings       | Active 12 В· Reserved 2 В· Sold 4   |
| - Transactions   |-----------------------------------|
| - Profile        | Listings needing action            |
|                  | Recent contact clicks              |
+------------------+-----------------------------------+
```

Dashboard should prioritize action:

1. Add listing.
2. Update reserved/sold status.
3. Send confirmation link.
4. Review contact clicks.

Avoid analytics-heavy widgets before usage exists.

### 6. Create Listing `/dashboard/listings/new`

Purpose: make sellers willing to list 5 real items quickly.

Form structure:

```text
Step 1: Photos
Step 2: Model identity
Step 3: Condition
Step 4: Price / trade
Step 5: Contact and publish
```

This can be one page with grouped sections. Do not use a multi-step wizard unless the form becomes too long on mobile.

Field hierarchy:

- Photos first, because marketplace trust is photo-led.
- Brand, model, scale second.
- Condition and defects before price.
- Contact method last.

Visible labels required. Never use placeholder-only labels.

Success state:

- `Listing published`
- Show public URL.
- Primary action: `Copy listing link`
- Secondary action: `Create another listing`

Partial failure:

- If upload fails, listing remains draft.
- User sees: `Photos did not finish uploading. Your listing is saved as a draft.`
- Primary action: `Retry upload`

### 7. Transaction Confirmation `/transactions/confirm/:token`

Purpose: make buyer confirmation feel safe and low-friction.

Layout:

```text
+----------------------------------+
| Confirm completed transaction    |
| Listing title                    |
| Seller name                      |
| Transaction type                 |
|                                  |
| [Confirm transaction]            |
| This confirms the transaction    |
| happened. It does not rate the   |
| seller or handle payment issues. |
+----------------------------------+
```

Do not overclaim. This confirmation is not a review and not buyer protection.

States:

- Valid pending token: show confirm action.
- Already confirmed: show `Already confirmed`.
- Expired token: show `This confirmation link expired`.
- Invalid token: show `This link is not valid`.

## User Journey Storyboard

### Seller: first listing

| Step | User does | User feels | UI support |
|---|---|---|---|
| 1 | Opens site | skeptical | marketplace shows real inventory, not marketing fluff |
| 2 | Creates profile | cautious | short form, optional identity links |
| 3 | Adds photos and details | busy | grouped form, visible required fields |
| 4 | Publishes listing | wants proof it worked | public URL and copy action |
| 5 | Shares link to FB/LINE | wants less repeated work | listing detail answers common buyer questions |
| 6 | Marks sold | wants credit | confirmation link creates transaction history |

### Buyer: evaluate listing

| Step | User does | User feels | UI support |
|---|---|---|---|
| 1 | Searches model | goal-oriented | large search and relevant filters |
| 2 | Opens listing | evaluating risk | photos, defects, price, seller trust visible |
| 3 | Checks seller | cautious | confirmed transaction count and profile history |
| 4 | Contacts seller | ready but alert | clear contact method, no hidden terms |
| 5 | Confirms transaction | wants low friction | one action, clear explanation |

## Responsive Rules

### Desktop >= 1024px

- Browse page uses left filter sidebar and 3-4 column listing grid.
- Listing detail uses two-column layout: gallery left, purchase/trust panel right.
- Dashboard uses persistent left navigation.

### Tablet 768-1023px

- Browse filters collapse into top filter row or drawer.
- Listing grid uses 2 columns.
- Listing detail stacks gallery above details, with sticky contact bar optional.

### Mobile <= 767px

- Browse has search at top, filter drawer, 1-column listing cards.
- Listing detail has photo gallery first, then price/status/contact, then seller trust.
- Contact action should be sticky at bottom only if it does not cover content.
- Touch targets minimum 44px.
- Filter chips must wrap cleanly.
- Long listing titles must not push status/action controls off-screen.

## Accessibility Requirements

- All inputs have visible labels.
- All buttons have text labels or accessible names.
- Listing cards are keyboard reachable.
- Filter drawer traps focus while open and closes with Escape.
- Color contrast minimum 4.5:1 for normal text.
- Status must not rely on color alone.
- Main landmarks: `header`, `nav`, `main`, `aside`, `footer`.
- Gallery thumbnails must have alt text or clear fallback.
- Confirmation page must announce success/error state.

## AI Slop Risk Review

Classifier: **APP UI with marketplace public pages**.

Hard rules:

- No generic SaaS hero.
- No 3-column feature grid.
- No decorative icon cards.
- No purple/blue gradient brand default.
- No dashboard card mosaic.
- No "trusted by collectors" claims without data.

Marketplace pages should use product inventory as the primary visual material. Photos of model cars carry the experience.

## Unresolved Design Decisions

### D1: Brand name and logo

Recommendation: use a neutral working name during MVP, such as `ModelTrade` or `Diecast Market`, and do not over-invest in logo design yet.

Why: the marketplace needs trust and inventory before brand polish matters.

### D2: Default marketplace segment

Recommendation: start UI filter presets with `Brand`, `Scale`, and `Listing mode`, not a hard-coded niche.

Why: the schema can support multiple segments, while the seed inventory determines which filters appear first.

### D3: Contact method

Recommendation: external contact first, with clear display of seller-provided method.

Why: in-platform chat creates moderation and notification complexity too early.

### D4: Seller trust level

Recommendation: show raw confirmed transaction count only.

Why: badges like "trusted" or "verified" overpromise and create liability before the platform has review/verification operations.

## Implementation Notes For Designers And Engineers

- Build marketplace home as a functional browse entry, not a marketing landing page.
- Use listing photos as the visual anchor.
- Put seller trust near the contact action on every listing detail page.
- Make defects and condition visible, not hidden behind accordions.
- Keep dashboards operational and compact.
- Write empty states before implementation. Do not ship `No items found`.
- Every mobile screen must be reviewed at 375px width before release.
