# Design: Model Car Marketplace

Generated by /office-hours on 2026-05-13
Status: DRAFT
Mode: Startup validation

## Problem Statement

Model car buyers and sellers currently use Facebook groups for buying, selling, and exchanging model cars. The behavior already exists, but the transaction environment is messy: posts are hard to search, item status is unclear, seller trust is difficult to verify, and transaction history is scattered across comments, profiles, and private messages.

The goal is to replace the Facebook group transaction behavior with a model-car-specific marketplace.

## Target User

The first target user is the frequent model car seller, proxy buyer, or bulk trader.

This user has repeated supply, sells or trades often, and benefits from building visible trust. Buyers matter, but the seller is the user most likely to create listings and bring inventory onto the platform.

## Demand Evidence

Current evidence:

- Model car buying, selling, and trading already happens inside Facebook groups.
- The main stated pain is trust: buyers check whether sellers have transaction records before dealing with them.
- The current workaround is manual: buyers inspect seller history, group activity, comments, and past transaction traces.

This is evidence of existing behavior, not yet evidence of willingness to move to a new platform. The first validation task is to prove that sellers will list inventory somewhere outside Facebook and use that link in their existing sales flow.

## Status Quo

Today, transactions happen in Facebook groups.

Typical flow:

1. Seller posts a model car listing in a group.
2. Buyer sees the post, comments, or sends a private message.
3. Buyer checks whether the seller appears trustworthy.
4. Trust is inferred from visible transaction history, group activity, comments, and reputation.
5. Payment, delivery, and final confirmation happen outside any structured system.

The status quo works because the community is already there. It fails because transaction information is not structured and trust is hard to verify.

## Premises

1. The first version should aim to replace Facebook group trading behavior, not merely assist it.
2. Facebook groups can still be used as a distribution channel during cold start.
3. The first supply-side users should be frequent sellers, proxy buyers, and bulk traders.
4. The MVP must include product listings and transaction history confirmation.
5. The first version should not handle payment, logistics, escrow, or dispute arbitration.
6. The product must feel model-car-native, not like a generic secondhand marketplace.

## Approaches Considered

### Approach A: Model Car Listing Board + Transaction Confirmation

Build a narrow buy/sell/trade board with structured model car fields and post-transaction confirmation.

Effort: S/M
Risk: Medium

Pros:

- Fastest path to validate whether structured listings are better than Facebook posts.
- Keeps the product small while still supporting real transaction behavior.
- Avoids premature complexity around payments and disputes.

Cons:

- May feel too similar to a classified board.
- Buyer behavior may remain mostly on Facebook.
- Less compelling as a standalone destination.

### Approach B: Full Model Car Marketplace

Build a dedicated marketplace with listings, search, filters, seller profiles, transaction states, buyer confirmation, and seller transaction records. Payment and logistics stay outside the platform in the first version.

Effort: L/XL
Risk: High

Pros:

- Closest to the long-term goal of replacing Facebook group trading.
- Gives buyers a real destination for browsing available models.
- Creates a foundation for future payments, logistics, ratings, and dispute support.

Cons:

- Cold start risk is high: an empty marketplace has little value.
- More features must work before the product feels credible.
- Building too much before proving seller migration could waste time.

### Approach C: Seller Page + Tradable Garage

Give each seller a public inventory page showing available, tradable, reserved, and sold model cars. Sellers share this page into Facebook and LINE groups while transaction history builds in the background.

Effort: M
Risk: Medium

Pros:

- Easier cold start because each seller gets an immediately useful storefront.
- Fits current behavior because sellers can still distribute links through Facebook.
- Builds toward marketplace liquidity without requiring a full buyer destination on day one.

Cons:

- Less direct as a Facebook replacement.
- Marketplace browsing may come later.
- Depends heavily on seller motivation to maintain inventory.

## Recommended Approach

Chosen approach: **Approach B, Full Model Car Marketplace**.

This matches the stated goal: replace the transaction behavior currently happening in Facebook groups.

The recommendation has one constraint: the first release should be a marketplace without platform-managed payment, logistics, escrow, or arbitration. Those systems are expensive operationally and legally, and they are not required to validate whether model car sellers and buyers want a dedicated transaction environment.

The practical version of Approach B should start with marketplace surfaces, but scope the first release tightly:

- Seller account and public seller profile
- Product listing creation
- Model-car-specific fields
- Browse and search
- Listing status: available, reserved, sold, trade-only
- Seller transaction count
- Buyer-confirmed transaction records
- External contact method or lightweight in-platform inquiry

## Core Product Requirements

### Listings

Each listing should include:

- Brand
- Model name
- Scale
- Series or release line
- Condition of car
- Condition of box
- Missing parts or defects
- Photos
- Price
- Trade availability
- Location or shipping preference
- Listing status

### Seller Profile

Each seller profile should show:

- Display name
- Seller bio
- Active listings
- Sold listings
- Confirmed transaction count
- Recent confirmed transactions
- Optional external social proof, such as Facebook profile or group identity

### Transaction Confirmation

After a sale or exchange:

1. Seller marks a listing as sold or traded.
2. Seller enters the buyer account or sends a confirmation link.
3. Buyer confirms the transaction happened.
4. Seller transaction count increases.
5. The listing moves into sold history.

The first version proves only that the seller has completed confirmed transactions. It should not yet score packaging quality, response speed, or item accuracy.

## What Not To Build First

Do not build these in the first release:

- Platform payment
- Escrow
- Logistics integration
- Automated dispute handling
- Auction mechanics
- Complex ratings
- Social feed
- Recommendation engine
- Mobile app

These are later-stage features. The MVP must first prove inventory, buyer search, and seller trust.

## Success Criteria

The first version is working if:

- 20 frequent sellers create profiles.
- 100 real listings are posted.
- At least 30 percent of listings come from sellers who post more than one item.
- At least 10 transactions are confirmed by both sides.
- Sellers share marketplace links back into Facebook groups or private chats.
- Buyers say the marketplace made it easier to decide whether a seller was trustworthy.

## Distribution Plan

Initial distribution should use the existing Facebook group behavior instead of fighting it immediately.

1. Recruit frequent sellers from model car Facebook groups.
2. Give them marketplace seller profiles.
3. Ask them to list inventory on the marketplace first.
4. Let them share marketplace listing links back into Facebook posts.
5. Track whether buyers click, inquire, and complete transactions.

The marketplace should replace the transaction destination gradually, while Facebook acts as the first acquisition channel.

## Open Questions

- Which model car segment should launch first: diecast, plastic model kits, Tomica, Hot Wheels, Mini GT, or premium resin models?
- Should the first version support exchange, or only buy/sell?
- Should buyer and seller identities be tied to Facebook profiles at launch?
- Should inquiries happen inside the platform or through external contact links?
- What minimum proof is required for a buyer-confirmed transaction?

## The Assignment

Before building the full marketplace, find 10 frequent model car sellers and ask each one:

1. Would you list 5 real items on a dedicated model car marketplace this week?
2. Would you share those marketplace links back into Facebook groups?
3. What information must be visible before a buyer trusts you?
4. What would make you stop using Facebook groups for the transaction itself?

If fewer than 5 sellers agree to list real inventory, the marketplace should be narrowed to seller pages or transaction records before building the full platform.

## What I Noticed

- You started broad with "模型車交流平台，可以買賣也可以交換", but the strongest pain you named was trust, not community.
- When challenged on whether this should merely support Facebook groups, you pushed back: "替待在社團上交易的行為". That changes the product from a trust add-on into a marketplace replacement.
- You chose the full marketplace approach even after seeing the cold-start risk. That is a valid founder choice, but it raises the validation bar: you now need seller commitment, not just buyer complaints.
