## Executive Decision

Build the MVP as a boring, web-first marketplace:

- Next.js App Router
- TypeScript
- Supabase Auth
- Supabase Postgres
- Supabase Storage
- Row Level Security on every public table
- Server-rendered public pages
- Server Actions or route handlers for mutations
- No custom backend service until the product has real transaction volume

This keeps the first version deployable by one person while still giving enough structure for auth, image uploads, listing search, seller profiles, and transaction confirmation.

### Complexity Check

This plan will touch more than 8 files once implemented. That is expected for a greenfield app with auth, database, UI, storage, and tests.

The complexity smell is not file count. The real risk is building too many product surfaces before proving liquidity. Keep the architecture modular but do not split into microservices, background workers, or separate API apps.

## Architecture

### Recommended Stack

Use:

- **Frontend:** Next.js App Router + React Server Components
- **Language:** TypeScript
- **Styling:** Tailwind CSS or plain CSS modules
- **Auth:** Supabase Auth with email magic link first
- **Database:** Supabase Postgres
- **Storage:** Supabase Storage for listing photos
- **Validation:** Zod schemas shared by server actions and tests
- **Testing:** Vitest for unit tests, Playwright for E2E
- **Deployment:** Vercel for Next.js, Supabase hosted project for DB/Auth/Storage

Do not use a custom Express/Nest/Rails API for v1. It adds another service boundary without adding product value.

### Source Rationale

Official docs support this direction:

- Next.js App Router supports server-side data fetching in Server Components and form handling through Server Actions.
- Supabase Auth integrates with Postgres and RLS so authorization can be enforced at row level.
- Supabase Storage supports direct file uploads, with standard uploads recommended for small files and resumable upload for larger files.

## Data Model

Use UUID primary keys everywhere. Use `created_at`, `updated_at`, and soft visibility states instead of hard deletes for marketplace records.

### `profiles`

One row per authenticated user.

```sql
id uuid primary key references auth.users(id) on delete cascade
display_name text not null
slug text unique not null
bio text
location text
facebook_url text
line_id text
avatar_url text
created_at timestamptz not null default now()
updated_at timestamptz not null default now()
```

Indexes:

- unique `slug`
- `created_at`

Rules:

- Public can read profiles.
- Authenticated user can insert/update only their own profile.

### `listings`

```sql
id uuid primary key default gen_random_uuid()
seller_id uuid not null references profiles(id) on delete restrict
title text not null
brand text not null
model_name text not null
scale text
series text
car_condition text not null
box_condition text
defects text
price numeric(12,2)
currency text not null default 'TWD'
trade_available boolean not null default false
location text
delivery_preference text
contact_method text not null
status text not null
visibility text not null default 'public'
created_at timestamptz not null default now()
updated_at timestamptz not null default now()
```

Allowed `status`:

- `available`
- `reserved`
- `sold`
- `trade_only`

Allowed `visibility`:

- `draft`
- `public`
- `archived`

Indexes:

- `(visibility, status, created_at desc)`
- `(seller_id, status)`
- `(brand)`
- `(scale)`
- `to_tsvector('simple', title || ' ' || brand || ' ' || model_name || ' ' || coalesce(series, ''))`

Rules:

- Public can read `visibility = 'public'`.
- Seller can create/update/archive their own listings.
- Seller cannot change `seller_id`.

### `listing_photos`

```sql
id uuid primary key default gen_random_uuid()
listing_id uuid not null references listings(id) on delete cascade
seller_id uuid not null references profiles(id) on delete restrict
storage_path text not null
alt_text text
sort_order int not null default 0
created_at timestamptz not null default now()
```

Indexes:

- `(listing_id, sort_order)`
- `(seller_id)`

Rules:

- Public can read photos for public listings.
- Seller can add/delete photos only for their own listings.
- Store `seller_id` redundantly to make RLS simpler and faster.

### `transactions`

```sql
id uuid primary key default gen_random_uuid()
listing_id uuid not null references listings(id) on delete restrict
seller_id uuid not null references profiles(id) on delete restrict
buyer_id uuid references profiles(id) on delete set null
buyer_email text
transaction_type text not null
status text not null default 'pending_buyer_confirmation'
confirmation_token_hash text unique
confirmed_at timestamptz
expires_at timestamptz not null
created_at timestamptz not null default now()
updated_at timestamptz not null default now()
```

Allowed `transaction_type`:

- `sale`
- `exchange`

Allowed `status`:

- `pending_buyer_confirmation`
- `confirmed`
- `expired`
- `cancelled`

Rules:

- Seller can create a transaction for their own listing.
- Public cannot read pending transaction details.
- Public seller profile can show aggregate confirmed transaction count.
- Buyer confirmation uses a one-time token and never stores the raw token.

### `listing_events`

Basic analytics. Keep it simple.

```sql
id uuid primary key default gen_random_uuid()
listing_id uuid references listings(id) on delete cascade
seller_id uuid references profiles(id) on delete set null
event_type text not null
session_id text
referrer text
created_at timestamptz not null default now()
```

Allowed `event_type`:

- `listing_view`
- `seller_profile_view`
- `contact_click`
- `share_click`

Rules:

- App can insert events.
- Sellers can read aggregate counts for their own listings.
- Do not expose raw event rows publicly.

## State Machines

### Listing State

```text
draft
  |
  | publish
  v
available <------+
  |              |
  | reserve      | unreserve
  v              |
reserved --------+
  |
  | mark sold / exchanged
  v
sold
  |
  | archive
  v
archived
```

Rules:

- Only seller can transition their own listing.
- `sold` should not be editable back to `available` after a confirmed transaction exists.
- `trade_only` is better as a listing status only if it excludes sale. If mixed sale/trade is allowed, make it a boolean or listing mode.

Recommendation: replace `trade_only` as a status with `listing_mode`.

```text
status: available | reserved | sold | archived | draft
listing_mode: sale | trade | sale_or_trade
```

This avoids invalid states like `status = trade_only` and `trade_available = false`.

### Transaction State

```text
pending_buyer_confirmation
  |             |
  | confirm     | expire / cancel
  v             v
confirmed     expired / cancelled
```

Rules:

- Confirmation token can be used once.
- Expired tokens should show a clear error.
- Confirming a transaction should be idempotent: if it is already confirmed, return success and do not increment counts twice.
- Confirmed transaction count should be computed from `transactions where status = 'confirmed'`, not manually incremented.

## Request Flow

### Seller Creates Listing

```text
Seller form
  |
  v
Server Action: createListing(formData)
  |
  +-- validate required fields with Zod
  +-- verify authenticated user
  +-- verify profile exists
  +-- insert listing as draft/public
  +-- upload photos to seller/listing scoped path
  +-- insert listing_photos rows
  +-- revalidate listing pages
  |
  v
Redirect to listing detail or edit page
```

Failure handling:

- Validation errors return field-level messages.
- Upload failure should not create a half-published listing. Prefer create listing as `draft`, upload photos, then publish.
- If DB insert succeeds but photo upload fails, user sees draft recovery state.

### Buyer Opens Listing

```text
Public listing route
  |
  v
Server Component fetches listing + seller + photos
  |
  +-- only visibility = public
  +-- 404 for draft/archived/nonexistent listing
  +-- insert view event best-effort
  |
  v
Render listing detail
```

Failure handling:

- Missing listing: useful 404.
- Seller profile missing: listing should not render as orphaned.
- Photo unavailable: show fallback image slot, not broken layout.

### Seller Records Transaction

```text
Seller marks sold
  |
  v
Server Action: createTransaction(listingId, buyerEmail)
  |
  +-- verify seller owns listing
  +-- verify listing is available/reserved
  +-- set listing status = sold
  +-- create pending transaction
  +-- generate raw token, store hash only
  +-- send or display confirmation link
  |
  v
Buyer opens /transactions/confirm/:token
  |
  +-- hash token
  +-- find non-expired pending transaction
  +-- confirm once
  +-- show success
```

Failure handling:

- Expired link: clear error and ask seller to resend.
- Already confirmed: show success, no double count.
- Wrong seller: reject with 403.
- Listing already sold: do not create duplicate confirmed transaction.

## Auth And Authorization

Use Supabase Auth for identity.

RLS must be enabled on every table in the public schema. Supabase docs recommend explicit authentication checks such as `(select auth.uid()) is not null` and role-specific policies for performance and clarity.

Recommended policies:

- `profiles`: public select, owner insert/update.
- `listings`: public select for public listings, owner insert/update/archive.
- `listing_photos`: public select through public listings, owner insert/delete.
- `transactions`: seller insert/select own transactions, buyer confirm through server action only.
- `listing_events`: insert from app, aggregate read by seller only.

Never expose the Supabase service role key to the browser.

## Search

Start with Postgres search, not Algolia/Meilisearch.

MVP search:

- Keyword search across title, brand, model name, series.
- Filters for brand, scale, status, listing mode.
- Sort by created_at desc.

Upgrade only when:

- Search latency becomes visibly slow.
- Relevance quality blocks buyer discovery.
- Faceted search becomes central to usage.

## Image Upload Strategy

Use Supabase Storage.

Path convention:

```text
listing-photos/{seller_id}/{listing_id}/{photo_id}.{ext}
```

Rules:

- Max 8 photos per listing.
- Max 6 MB per photo for standard upload.
- Generate new object paths; do not overwrite existing objects.
- Validate content type and size before upload.
- Store only storage paths in DB.

If sellers upload larger files later, add resumable uploads. Do not build that first.

## Caching

Default:

- Public listing pages can be cached briefly.
- Seller dashboards must not be cached.
- Mutations must revalidate affected listing, seller, and browse pages.

Use framework cache invalidation after create/update/delete listing and after transaction confirmation.

## NOT In Scope

- Payment processing: deferred until real transaction volume and risk justify it.
- Escrow: operational burden too high for MVP.
- Shipping integration: not needed to validate listings and trust.
- In-platform chat: use external contact first to avoid moderation burden.
- Ratings: confirmed transaction count is the first trust signal.
- Auctions and bidding: adds state complexity without proving core liquidity.
- Native mobile app: web links are better for Facebook/LINE distribution.
- Recommendation engine: search and filters are enough for MVP.
- Admin console: can start with direct database/admin dashboard for seed cohort.

## Architecture Review Findings

### 1. Listing state model is currently overloaded

Severity: P1
Confidence: 8/10

The product spec mixes transaction state and listing mode:

- `status = available | reserved | sold | trade only`
- `trade_available = boolean`

This creates invalid combinations.

Recommendation: use separate fields:

- `status = draft | available | reserved | sold | archived`
- `listing_mode = sale | trade | sale_or_trade`

Decision: adopt this in implementation.

### 2. Transaction confirmation needs token security and idempotency

Severity: P1
Confidence: 8/10

The spec says seller sends a confirmation link, but does not define token handling. Raw tokens in the DB are avoidable risk.

Recommendation:

- Generate random token.
- Store hash only.
- Add `expires_at`.
- Confirm in a transaction.
- Return success if already confirmed.

Decision: adopt this in implementation.

### 3. Photo upload can create partial records

Severity: P2
Confidence: 7/10

If listing creation and photo upload are not sequenced carefully, users can end up with public listings missing photos or orphaned storage objects.

Recommendation:

- Create listing as draft.
- Upload photos.
- Insert photo rows.
- Publish only after required photos succeed.
- On failure, keep draft and show recovery UI.

Decision: adopt this in implementation.

## Open Decisions

### D1: Buyer account for contact

Recommendation: buyers do not need accounts to contact sellers in MVP.

Reason: requiring buyer accounts too early reduces marketplace liquidity. Transaction confirmation can require email-token confirmation first, then buyer account later.

Decision: no buyer account required for contact.

### D2: Buyer account for transaction confirmation

Recommendation: do not require buyer account for first confirmation. Use email/token confirmation.

Reason: the trust signal needed is "another party confirmed this transaction happened." Requiring full signup adds friction before the network has value.

Decision: email/token confirmation for MVP.

### D3: Exchange support

Recommendation: support exchange from day one through `listing_mode`, but do not build matching.

Reason: exchange is part of the original product idea, and the schema cost is small. Matching algorithms and trade proposals are not in scope.

Decision: `listing_mode = sale | trade | sale_or_trade`.

### D4: Facebook identity

Recommendation: optional Facebook URL, not required.

Reason: requiring Facebook identity conflicts with the goal of replacing Facebook, but optional social proof helps early trust.

Decision: optional.

### D5: First model segment

Recommendation: start schema generic across diecast/model cars, but seed UI filters with the first seller cohort's brands and scales.

Reason: hard-coding one segment too early risks unnecessary rework. But the UI should not feel generic.

Decision: generic schema, cohort-specific filter presets.
